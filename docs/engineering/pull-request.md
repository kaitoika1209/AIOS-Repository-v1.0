# Pull Request Guidelines

## Purpose

Pull Requests provide an opportunity to review changes before they become part of the project.

---

# One Purpose Per Pull Request

A pull request should solve one problem.

Avoid combining unrelated features.

---

# Pull Request Template

Every Pull Request should include:

- Summary
- Motivation
- Changes
- Testing
- Related Issue

---

# Review Requirements

Before requesting review:

- Code builds successfully.
- Tests pass.
- Documentation is updated if required.
- No debugging code remains.

---

# Keep Pull Requests Small

Prefer pull requests that can be reviewed in less than 30 minutes.

Large pull requests should be divided into smaller changes whenever possible.

---

# Merge Strategy

Squash and Merge is the default strategy unless preserving commit history is required.
